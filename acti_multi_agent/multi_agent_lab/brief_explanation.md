# Brief Explanation

**Agent Roles & Workflow**: Three agents execute sequentially in a chain: Agent 1 (Literature Scout) surveys six key papers on AI model collapse and data pollution, establishing the entropy-based problem framing; Agent 2 (Quantitative Analyst) receives Agent 1's output and formalizes the Authenticity Loss Function L_auth = λ₁·D_KL + λ₂·ΔH + λ₃·V_tail with six measurable metrics; Agent 3 (Solution Architect) receives both prior outputs plus real CampusGo project data and maps platform features to the quantitative framework through a four-layer authenticity stack. Each agent's output is passed as the user message to the next agent via the `agent_run()` helper function.

**Prompt Design & Iterations**: Four iterations refined the system — v1 used basic role descriptions but produced unstructured output; v2 added strict markdown headers, improving structure but leaving inter-agent coherence weak; v3 injected concrete numerical anchors (3.119/3.883 bits entropy gap, Var(X_jⁿ) = σ²(1+n/M)) and per-agent word limits (800/600/800); v4 extracted behavioral rules into a separate YAML file (`agent_rules.yaml`) and introduced `agent_run()` and `format_rules_for_prompt()` helpers, making the system modular and maintainable without touching code.

**What Worked & What Changed**: The biggest improvement from v3 was injecting specific numerical values and real project data, which forced agents to ground output in concrete evidence and created natural cross-references. The v4 refactor — separating rules into YAML and wrapping API calls in `agent_run()` — did not change output quality but made the system significantly easier to iterate on: rules can now be tweaked in `agent_rules.yaml` without modifying any Python code, mirroring the pattern taught in the course's `04_rules.py` lab.
